    Identity & Credential Structure

    The system's decentralized identity layer is built upon deterministic cryptographic key derivation (via BIP39 and BIP47), anchored by a secure naming and credentialing system. Each participant in the network is assigned a unique, verifiable identity composed of:

    • Mnemonic Seed (BIP39)
    • Public/Private Keypair (seckp256k1)
    • Sovereign Name Identifier (sfname)
    • Paycode (BIP47-compatible)
    • Signed Contact Credential (optional)

    Wallet Identity

    When a user imports or generates a wallet:

    • A 12-24 word mnemonic is used to deterministically derive the master seed.
    • This seed is transformed into a root keypair using seckp256k1.
    • The sfname is registered as a globally unique human-readable handle.
    • A paycode is derived from the public key using the BIP47 protocol.

    Example Wallet Identity Object:

    {
  "mnemonic": "abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon about",
  "master_seed": "0x...",
  "keypair": {
    "public_key": "0x0279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798",
    "private_key": "0x...",
    "compressed": true
  },
  "sfname": "alice.wallet",
  "paycode": "PM8TJXuZ63i8AuSBLVUuqVQt3HFPwz2KGPNWJ3K6WrEZFqeJVZBHU4CQVYLK8He",
  "created_at": "2025-08-18T10:00:00Z"
}

    Contact Credential

    A Contact Credential is a signed payload representing the linking of one wallet's public identity to another. It enables:

    • Mutual key discovery and relationship acknowledgment
    • Session recovery and replay prevention
    • In-message authorization and payment signaling

    Each Contact Credential contains:

    • Issuer sfname and public key
    • Recipient sfname and public key
    • Issuance timestamp
    • seckp256k1 signature of the payload

    Example Contact Credential Payload:

    {
  "type": "ContactCredential",
  "version": "1.0",
  "issuer": {
    "sfname": "alice.wallet",
    "pubkey": "0x0279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798"
  },
  "recipient": {
    "sfname": "bob.wallet",
    "pubkey": "0x03B9A7E9C3D4E5F6A7B8C9D0E1F2A3B4C5D6E7F8A9B0C1D2E3F4A5B6C7D8E9F0"
  },
  "issued_at": "2025-08-18T10:00:00Z",
  "metadata": {
    "relationship": "friend",
    "notes": "Regular trading partner"
  },
  "signature": "0x..."
}

    Authority Signature (Nym System)

    To validate identity origin and ensure protection against impersonation, each identity is optionally backed by a Nym Authority signature. This signature:

    • Confirms linkage to a verifiable messaging node (signal relayer)
    • Can be validated offline
    • Enables private routing enforcement policies

    Nym Authority Proof Structure:

    {
  "authority_id": "nym-authority-001",
  "identity_proof": {
    "sfname": "alice.wallet",
    "nym_address": "nym://alice@relay.nymtech.net",
    "proof_type": "nym_binding"
  },
  "issued_at": "2025-08-18T10:00:00Z",
  "valid_until": "2026-08-18T10:00:00Z",
  "signature": "0x..."
}

    Contact Graph Representation

    Internally, all contact credentials form a directed acyclic graph (DAG) stored in the user's local WalletSDK cache and replicated selectively via encrypted backup channels.

    Each node = sfname Each edge = Signed ContactCredential

    Revocation & Expiry

    Credentials may include TTL (time-to-live) headers or be manually revoked. WalletSDK will prune expired or revoked credentials from the graph to maintain consistency.

    Revocation Object:

    {
  "type": "CredentialRevocation",
  "credential_id": "cred_12345",
  "revoked_by": "alice.wallet",
  "revoked_at": "2025-08-18T10:00:00Z",
  "reason": "contact_terminated",
  "signature": "0x..."
}

    Paycode Write and Verification (DeFi Anchor)

    Upon successful POST /identity/register, the system performs the following:

    Write to DeFi Signaling Contract:

    // Smart contract call
await signalingContract.registerIdentity(
  sfname,
  paycode,
  pubkey,
  signature
);

    This event acts as the canonical record of identity registration.

    SDK Verification Flow:

    // Verify on-chain registration
const events = await signalingContract.queryFilter(
  signalingContract.filters.IdentityRegistered(sfname)
);

if (events.length > 0) {
  // Registration confirmed
  updateLocalState({ identityConfirmed: true });
}

    This ensures the on-chain registration was successful and links: sfname → paycode → pubkey.

    Fallback Logic: If verification fails (e.g. no event found), the SDK:

    • Displays a retry UI for re-registration
    • Logs event in localAuditTrail()
    • Flags the state as identityNotConfirmed

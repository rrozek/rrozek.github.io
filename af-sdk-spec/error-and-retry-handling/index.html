    Error and Retry Handling

    This section outlines how the WalletSDK and signaling infrastructure handle delivery failures, decryption errors, credential mismatches, and network interruptions to preserve integrity, confidentiality, and liveness of message delivery and payment execution.

    Error Categories

    Category Description Examples
    Network Errors Connectivity issues Timeout, DNS failure, connection reset
    Authentication Errors Token/session issues Expired token, invalid signature
    Decryption Errors Cryptographic failures Invalid key, corrupted ciphertext
    Protocol Errors Message format issues Invalid payload structure, version mismatch
    Resource Errors System limitations Rate limiting, storage full
    Severity Impact Response
    Critical Service unavailable Immediate alert, manual intervention
    High Feature degradation Automatic retry with fallback
    Medium Partial functionality Degraded mode operation
    Low Minor inconvenience Log and continue

    Retry Policy

    • Transient Network Errors: Retries every 3 seconds, max 5 attempts
    • Decryption Failures: No retry, log for manual review
    • Token Expiry: Prompt re-authentication before reconnect
    • Invalid Signature or Credential: Fail-fast and flag signal

    Signal Backup and Recovery Protocol

    Signal reliability is critical in a decentralized, peer-to-peer wallet environment. This protocol ensures message delivery, backup, and recovery across various fault domains (offline peers, smart contract congestion, temporary Nym/Notary downtime).

    Each outbound signal is handled using a multi-path broadcast strategy combined with local caching:

    Path Description Reliability Privacy
    Primary Nym Route Onion routing through mixnet High Maximum
    DeFi Beacon Backup On-chain signal storage Maximum Medium
    Direct P2P Fallback Peer-to-peer connection Variable Low
    Encrypted Local Cache Device storage backup High Maximum

    If a signal fails to reach its destination via one method, it is retried via alternate paths automatically using a priority system.

    Outbox & Retry Queue (WalletSDK)

    The WalletSDK maintains a local queue of signals that:

    • Were not acknowledged via ack
    • Were not retrieved within ACK_TIMEOUT

    Retry Logic Example

    Retries prioritize:

    1. Nym if online
    2. DeFi beacon path
    3. Direct SDK peer-to-peer if contact is reachable
    class SignalRetryManager {
  private retryQueue: SignalRetryItem[] = [];

  async enqueueForRetry(signal: SignalMessage, error: Error): Promise<void> {
    const retryItem: SignalRetryItem = {
      signal,
      error,
      attempts: 0,
      nextRetryAt: this.calculateNextRetryTime(0),
      strategies: this.getAvailableStrategies(signal)
    };

    this.retryQueue.push(retryItem);
    await this.persistRetryQueue();
  }

  private calculateNextRetryTime(attempt: number): Date {
    // Exponential backoff: 3s, 9s, 27s, 81s, 243s
    const delayMs = Math.pow(3, attempt) * 1000;
    return new Date(Date.now() + delayMs);
  }
}

    Signal Expiry & Garbage Collection

    Signals are tagged with:

    • Creation timestamp
    • TTL (time-to-live) value
    • Priority level
    • Retry count

    WalletSDK periodically runs cleanup jobs that remove expired, unacknowledged signals from the outbox.

    interface SignalMetadata {
  id: string;
  createdAt: Date;
  ttl: number; // seconds
  priority: 'low' | 'normal' | 'high' | 'urgent';
  retryCount: number;
  maxRetries: number;
}

class SignalGarbageCollector {
  async collectExpiredSignals(): Promise<void> {
    const expiredSignals = await this.findExpiredSignals();
    for (const signal of expiredSignals) {
      await this.notifySender(signal);
      await this.removeFromOutbox(signal);
    }
  }

  private async findExpiredSignals(): Promise<SignalMetadata[]> {
    const now = Date.now();
    return this.signalStore.find({
      $where: (signal) => {
        const expiryTime = signal.createdAt.getTime() + (signal.ttl * 1000);
        return now > expiryTime;
      }
    });
  }
}

    DeFi-Based Redundancy

    All beaconed signals are also written on-chain using:

    function storeSignal(
    bytes32 signalId,
    address recipient,
    bytes calldata encryptedSignal,
    bytes calldata ephemeralPubKey
) external {
    // Store encrypted signal on-chain for redundancy
    signals[signalId] = Signal({
        recipient: recipient,
        encryptedData: encryptedSignal,
        ephemeralPubKey: ephemeralPubKey,
        timestamp: block.timestamp,
        isDelivered: false
    });
}

    Receiving wallets can:

    • Query smart contract logs for missed beacons
    • Use ephemeralPubKey to decrypt the signal retroactively
    • Sync missed message history post-reconnect

    Offline Peer Recovery

    If the recipient wallet is offline, upon next boot:

    1. SDK queries last known DeFi signal anchors
    2. Attempts ECDH decryption of beacons
    3. Triggers normal message receipt if match is found
    class OfflineRecoveryManager {
  async performRecovery(): Promise<void> {
    const lastKnownBlock = await this.getLastProcessedBlock();
    const missedSignals = await this.queryMissedSignals(lastKnownBlock);

    for (const signal of missedSignals) {
      try {
        const decryptedSignal = await this.attemptDecryption(signal);
        if (decryptedSignal) {
          await this.processDecryptedSignal(decryptedSignal);
        }
      } catch (error) {
        console.warn('Failed to decrypt missed signal:', error);
      }
    }
  }
}

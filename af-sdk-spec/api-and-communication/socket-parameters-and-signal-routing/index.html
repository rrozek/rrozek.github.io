    Socket Parameters & Signal Routing

    WebSocket-based signaling provides real-time, low-latency communication between verified identities. This section defines the socket initialization parameters, reconnection logic, signal packet formats, and dynamic routing strategies via Nym.

    WebSocket Initialization Parameters

    The client initiates a secure socket connection with the following headers:

    GET /signal HTTP/1.1
Host: api.wallet-signal.io
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Sec-WebSocket-Protocol: afsdk-signal-v1
Authorization: Bearer <sessionToken>

    Handshake Response Format

    Upon successful connection:

    {
  "type": "handshake_success",
  "session_id": "ws_session_12345",
  "user_sfname": "alice.wallet",
  "supported_protocols": ["afsdk-signal-v1"],
  "heartbeat_interval": 30000
}

    Signal Packet Structure

    Each signal packet is a signed envelope containing metadata and encrypted content.

    interface SignalPacket {
  type: 'signal' | 'ack' | 'error';
  id: string;
  timestamp: number;
  sender: string;
  recipient: string;
  routing: RoutingInfo;
  payload: string; // Base64 encrypted content
  signature: string; // Base64 signature
}

interface RoutingInfo {
  strategy: string;
  priority: 'low' | 'normal' | 'high' | 'urgent';
  ttl: number; // Time to live in seconds
  fallback_allowed: boolean;
}

    Dynamic Routing Logic

    Signal routing may dynamically select from available channels:

    Channel Description Latency Privacy Level
    Nym Route Onion routing through mixnet High Maximum
    Direct WebSocket Peer-to-peer connection Low Medium
    Beacon Broadcast On-chain signaling Variable High
    Backup Relay Encrypted storage High Maximum
    Hybrid Fallback Multi-channel approach Variable Adaptive

    Reconnection Policy

    If the socket connection fails or times out:

    • SDK attempts automatic reconnection every 3 seconds (up to 5 times)
    • SessionToken is reused unless expired
    • Event onReconnected is triggered for app-layer hooks
    class WebSocketManager {
  private reconnectAttempts = 0;
  private maxReconnectAttempts = 5;
  private reconnectInterval = 3000;

  private async attemptReconnect(): Promise<void> {
    if (this.reconnectAttempts >= this.maxReconnectAttempts) {
      throw new Error('Max reconnection attempts exceeded');
    }

    this.reconnectAttempts++;
    await new Promise(resolve => setTimeout(resolve, this.reconnectInterval));

    try {
      await this.connect();
      this.reconnectAttempts = 0;
      this.emit('reconnected');
    } catch (error) {
      this.attemptReconnect();
    }
  }
}

    Disconnection Scenarios

    Scenario Cause Recovery Action Timeout
    Network Loss Connectivity issues Auto-reconnect 30s
    Server Restart Maintenance Exponential backoff 60s
    Token Expiry Session timeout Token refresh + reconnect 5s
    Rate Limiting Too many requests Wait + retry 10s
    Protocol Error Invalid message format Manual reconnection N/A

    Fallback Routing Diagnostics

    When primary signal routing fails, fallback logic is triggered based on routing priority. The following diagnostics are collected to trace and debug these scenarios:

    • Attempted Route: Channel attempted (e.g., nym, direct)
    • Failure Reason: Reason reported by transport layer (e.g., timeout, signature invalid)
    • Next Route Selected: Channel selected for retry
    • Timestamped Logs: Each fallback transition is logged with UTC timestamps

    Example Fallback Log (JSON Format):

    {
  "timestamp": "2025-08-18T10:05:23Z",
  "signal_id": "sig_12345",
  "attempted_route": "nym",
  "failure_reason": "timeout",
  "next_route": "direct_websocket",
  "retry_count": 1,
  "total_latency_ms": 15000
}

    These logs are stored locally in encrypted form and optionally synced to secure diagnostic storage with user permission.

    Logging Strategy Levels

    Signal diagnostics are logged according to severity:

    Level Description Example Triggers
    ERROR Critical failures requiring immediate attention Connection completely lost
    WARN Potential issues that may affect performance High latency detected
    INFO Normal operational events Route switching successful
    DEBUG Detailed diagnostic information Strategy evaluation details
    TRACE Very detailed execution flow Individual packet routing

    Logging can be enabled with verbosity filters at runtime using SDK options:

    const loggingConfig = {
  level: 'info',
  filters: {
    include: ['routing', 'websocket'],
    exclude: ['heartbeat']
  },
  remoteSync: true,
  retentionDays: 30
};

    These diagnostics ensure observability for decentralized message delivery while preserving user privacy and maintaining system integrity.

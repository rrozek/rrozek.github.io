    Beaconing, Discovery, and Recovery

    Overview

    The key design goal is to establish a many-to-one mapping, where multiple users' payment codes deterministically map to the same beacon identifier, forming logical "buckets." This enables users to efficiently identify which encrypted messages in a shared inbox are intended for them — without needing to decrypt full payloads. The scheme combines asymmetric encryption, cryptographic tagging, and a lightweight header, all of which can be processed using only the recipient's private key.

    This design is particularly useful in decentralized messaging systems, wallet-based notification protocols, and off-chain asynchronous communication built on top of blockchain infrastructure.

    To enhance the OBPP-05 specification and support more scalable messaging in decentralized networks, we propose an algorithm that introduces shared beacon identifiers instead of assigning a unique one per user. This allows multiple recipients to monitor a single beacon address for relevant messages.

    In this context, a beacon acts as a detectable on-chain or off-chain marker that lets recipients infer whether a message might be intended for them — without revealing their identity until decryption.

    Key Benefits

    • Fewer unique scripts or notification addresses to monitor
    • Improved performance for lightweight clients using mechanisms like compact filters (e.g cfilters)
    • Compatibility with wallet-to-wallet notification protocols

    Proposed Algorithm (Enhancement of OBPP-05)

    Compute beacon_id

    Let:

    • p be the compressed public key from the recipient's payment code,
    • v be the 1-byte version of the sender's payment code (as per OBPP-05 compatibility).

    Steps:

    1. Compute a "chain code" from the recipient key:
    chain_code = HMAC-SHA512(p, "beacon-chain-code")
    1. Generate HMAC using sender version:
    hmac_result = HMAC-SHA512(chain_code, v)
    1. Truncate the HMAC to M bytes to produce a beacon identifier:
    beacon_id = hmac_result.substring(0, M)

    Then such a beacon_id can be used as a group identifier in smart_contract collection.

    Message Structure (Hybrid Encryption Mode)

    Each message includes the following fields:

    Field Description Size
    beacon_id Group identifier M bytes
    tag Encrypted nonce for filtering \~100 bytes
    hash_nonce SHA256 of nonce 32 bytes
    payload Encrypted paycode Variable
    metadata Routing information Variable

    The protocol uses a lightweight tag mechanism instead of hybrid AES. The payload is small (an encrypted paycode), so asymmetric encryption is sufficient and avoids unnecessary complexity. The tag allows recipients to filter inboxes efficiently before attempting payload decryption.

    Sender-Side Process

    1. Generate a 128-bit random nonce N.
    const nonce = crypto.getRandomValues(new Uint8Array(16));
    1. Compute:
    const tag = encryptWithPublicKey(nonce, recipientPublicKey);
const hashNonce = sha256(nonce);
    1. Encrypt the paycode payload directly with the recipient's public key (OBPP-5 style).
    2. Construct message:
    {
  "beacon_id": "beacon_12345",
  "tag": "encrypted_nonce_here",
  "hash_nonce": "sha256_hash_here",
  "payload": "encrypted_paycode_here",
  "timestamp": "2025-08-18T10:00:00Z"
}

    Recipient-Side Process

    For each message in the inbox:

    1. Decrypt tag with their private key to recover nonce N'
    2. Verify SHA256(N') == hash_nonce
    3. Match → proceed
    4. Mismatch → ignore message
    5. Decrypt payload using private key to recover the paycode.

    Pseudocode Example

    Sender:

    function createBeaconMessage(senderPaycode, recipientPaycode, message) {
  // Generate beacon ID
  const beaconId = computeBeaconId(recipientPaycode.publicKey, senderPaycode.version);

  // Create filtering tag
  const nonce = generateRandomNonce();
  const tag = encryptWithPublicKey(nonce, recipientPaycode.publicKey);
  const hashNonce = sha256(nonce);

  // Encrypt payload
  const encryptedPayload = encryptWithPublicKey(message, recipientPaycode.publicKey);

  return {
    beacon_id: beaconId,
    tag: tag,
    hash_nonce: hashNonce,
    payload: encryptedPayload
  };
}

    Recipient:

    function processBeaconMessage(message, privateKey) {
  // Try to decrypt tag
  try {
    const decryptedNonce = decryptWithPrivateKey(message.tag, privateKey);
    const computedHash = sha256(decryptedNonce);

    // Check if this message is for us
    if (computedHash !== message.hash_nonce) {
      return { relevant: false };
    }

    // Decrypt the actual payload
    const decryptedPayload = decryptWithPrivateKey(message.payload, privateKey);

    return {
      relevant: true,
      payload: decryptedPayload,
      beacon_id: message.beacon_id
    };
  } catch (error) {
    return { relevant: false };
  }
}

    End-to-End Scenario: Bob Sends Notification to Alice

    Actors:

    • Alice: Public key PK_A, Private key SK_A
    • Bob: Wants to send Alice a private notification: "Bob_pcode" (payload)

    Bob's Process:

    1. Bob Generates Random nonce (16 bytes)

    2. Bob Prepares the Header:

      • tag = EncryptWithPublicKey(N, PK_A)
      • hash_nonce = SHA256(N)

      Example:

      {
  "tag": "encrypted_nonce_data",
  "hash_nonce": "a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3"
}
      3. Bob Encrypts the payload 4. Bob Constructs the Message:

      {
  "beacon_id": "beacon_alice_group",
  "tag": "encrypted_nonce_data",
  "hash_nonce": "a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3",
  "payload": "encrypted_bob_paycode"
}

    Alice's Process:

    1. Decrypt the tag:
    2. N' = DecryptWithPrivateKey(tag, SK_A)
    3. Hash the result:
    4. hash_check = SHA256(N')
    5. Compare:
    6. If hash_check != hash_nonce, discard the message
    7. Otherwise, continue
    8. Decrypt the payload:
    9. If match → decrypts payload with SK_A → obtains Bob's paycode
    10. Result: Bob_pcode

    Summary

    • Simplicity: avoids the unnecessary AES layer
    • Efficiency: one small asymmetric decryption per message for filtering
    • Compatibility: reuses OBPP-5 signal encryption
    • Security: only the intended recipient can recognize and decrypt the message

    Advantages

    • Scalability: Each message requires only one small asymmetric decryption of the tag for filtering, making the system efficient even for large shared inboxes
    • Security: Only the intended recipient can validate and decrypt the payload; the hash_nonce check ensures tags cannot be forged
    • Simplicity: Eliminates the complexity of hybrid AES schemes — the entire mechanism relies solely on well-understood public-key cryptography
    • Robustness: The tag+hash mechanism prevents false positives and ensures that recipients never waste resources attempting to decrypt irrelevant payloads

    Potential Challenges & Mitigations

    Performance in Very Large Inboxes

    Challenge: Repeated asymmetric tag decryptions may still become costly.

    Mitigation: Use elliptic-curve-based encryption (e.g., ECIES) for faster operations, or batch-process tag checks in parallel.

    Header Size

    Challenge: Even with just tag + hash, the header introduces overhead (≈100-150 bytes with ECC).

    Mitigation: Optimize tag length or use compact encodings to minimize bandwidth/storage impact.

    Denial-of-Service (DoS)

    Challenge: An attacker could flood inboxes with fake messages, forcing recipients to decrypt many tags.

    Mitigation: Add inbox-level rate-limiting, lightweight metadata filtering, or reputation-based throttling before attempting asymmetric decryption.

    Alternative Approaches

    Key Derivation Tags

    Instead of encrypting a nonce, derive a tag from the recipient's public key using a key derivation function (KDF). For example:

    T = KDF(PK_i, sender_secret)

    The recipient verifies the tag using their private key and the sender's public key. This avoids an extra encryption but requires careful KDF design to prevent key leakage.

    Backup Smart Contract System

    Backup smart contract provides a secure, privacy-preserving way for wallets to store and recover lists of counterparty paycodes.

    Main Design Goals:

    • Unlinkability: The backup inbox must not be traceable to the user's public payment code
    • Write Protection: Only the owner wallet can write to their backup inbox
    • Privacy & Security: Data is encrypted; no one else can read or write to the inbox

    Use Case:

    1. Alice decrypts notifications from the beacon
    2. She gets Bob's paycode as a signal sender
    3. Alice encrypts Bob's paycode and appends it to her backup inbox
    4. Later, if Alice restores her wallet from seed, she derives backup private key again, reads her encrypted inbox, and decrypts her counterparty list
